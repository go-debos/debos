# debos - Debian OS image builder

## Synopsis

```
debos [options] <recipe file in YAML>
debos [--help]
```

Application Options:

```
  -b, --fakemachine-backend=[auto|kvm|qemu] Fakemachine backend to use (default: auto)
      --artifactdir=                        Directory for packed archives and ostree repositories (default: current directory)
  -t, --template-var=                       Template variables (use -t VARIABLE:VALUE syntax)
      --debug-shell                         Fall into interactive shell on error
  -s, --shell=                              Redefine interactive shell binary (default: bash) (default: /bin/bash)
      --scratchsize=                        Size of disk-backed scratch space (parsed with human-readable suffix; assumed bytes if no suffix)
  -c, --cpus=                               Number of CPUs to use for build VM (default: 2)
  -m, --memory=                             Amount of memory for build VM (parsed with human-readable suffix; assumed bytes if no suffix) (default: 2Gb)
      --show-boot                           Show boot/console messages from the fakemachine
  -e, --environ-var=                        Environment variables (use -e VARIABLE:VALUE syntax)
  -v, --verbose                             Verbose output
      --print-recipe                        Print the final recipe
      --dry-run                             Check the final recipe and verify all actions without executing them
      --disable-fakemachine                 Do not use fakemachine
      --version                             Print debos version
```

## Description

debos is a tool to make the creation of various Debian-based OS images
simpler. While most other tools focus on specific use-cases, debos is
designed to be a toolchain making common actions trivial while providing
enough rope to do whatever tweaking which might be required behind the
scenes.

debos expects a YAML file, called a recipe, as its input. The recipe is
pre-processed by the Golang [template engine](https://pkg.go.dev/text/template) and describes
a list of actions which debos runs sequentially.
Actions are designed to be self-contained and independent of each other.

The recipe syntax and file format and templating engine are described in the [Recipe
syntax](#recipe-syntax) section. Each available action, with its properties and a worked
example, is documented in the [Actions](#actions) section. The variables debos exports
to the build environment and the values for the `origin` property are described in the
[Environment variables](#environment-variables) section.

The reference documentation in this README is generated by hand and may lag
behind the code. The authoritative, automatically generated action,
documentation is published on the
[godoc documentation](https://godoc.org/github.com/go-debos/debos/actions).
When in doubt, treat the [godoc documentation](https://godoc.org/github.com/go-debos/debos/actions)
as the most up-to-date source.

## Installation (Docker container)

An official debos container is available:
```
docker pull godebos/debos
```

Debos needs virtualization to be enabled on the host and shared with the container.

Check that `kvm` is enabled and writable by the user running the docker container by running `ls /dev/kvm`

### Container usage

> This container should be used as an executable, i.e. there is no need to add `debos` after `godebos/debos`.

To build `recipe.yaml`:
```
cd <PATH_TO_RECIPE_DIR>
docker run --rm -it --device /dev/kvm --user $(id -u) --workdir /recipes --mount "type=bind,source=$(pwd),destination=/recipes" --security-opt label=disable godebos/debos <RECIPE.yaml>
```

If debos fails to run the KVM fakemachine backend and the `/dev/kvm` device exists on your host, you may need to add the owning group of the device as a supplementary group of the container. This will work if `ls -l /dev/kvm` indicates that the owning group has read-write access to the device. Adding the supplementary group may be unsafe depending on the owning group of `/dev/kvm`, but it could be required depending on your login provider. To add the group, add `--group-add "$(stat -c '%g' /dev/kvm)"` to your `docker run` command before `godebos/debos`. See [Docker run reference -- Additional Groups](https://docs.docker.com/engine/reference/run/#additional-groups) for more information.

### Container build

To build the debos container image from current git branch:
```
docker build -t godebos/debos .
```

## Using debos in GitHub Actions

debos can be run in GitHub Actions using the official container with KVM support
for isolated and reproducible builds. The `--fakemachine-backend=kvm` option is
specified to ensure KVM is used as expected:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    container:
      image: ghcr.io/go-debos/debos:main
      options: --device=/dev/kvm
    steps:
      - uses: actions/checkout@v4
      - run: debos --fakemachine-backend=kvm --print-recipe recipe.yaml
```

## Simple example

The following example will create an arm64 image, install several
packages in it, change the file `/etc/hostname` to `debian` and finally
make a tarball of the complete system.

```yaml
{{- $image := or .image "debian.tgz" -}}

architecture: arm64

actions:
  - action: debootstrap
    suite: trixie
    components:
      - main
      - non-free-firmware
    mirror: https://deb.debian.org/debian
    variant: minbase

  - action: apt
    packages:
      - sudo
      - openssh-server
      - adduser
      - systemd-sysv
      - firmware-linux

  - action: run
    chroot: true
    command: echo debian > /etc/hostname

  - action: pack
    file: {{ $image }}
    compression: gz
```

To run it, create a file named `example.yaml` and run:

```bash
debos example.yaml
```

The final tarball will be named `debian.tgz`. If you would like to modify
the fileame, you can provide a different name for the variable image
like this:

```bash
debos -t image:"debian-arm64.tgz" example.yaml
```

## Recipe syntax

A recipe is a YAML file. Before it is parsed as YAML it is run through the
Golang [template engine](https://pkg.go.dev/text/template), so a recipe is
effectively a template that produces YAML. A recipe has two parts:
    - a header
    - a list of actions.

Recipe comments should start with `#`.

### Recipe Header

The recipe header may contain the following properties:

- `architecture` -- target architecture (string). Mandatory; debos aborts if
  it is missing (e.g. `amd64` or `arm64`).
- `sectorsize` -- logical sector size in bytes (integer). Optional, defaults
  to `512`. Set this to `4096` for devices using 4K blocks such as some UFS or
  NVMe storage. It is consumed by the `image-partition` and `raw` actions.
- `actions` -- the list of actions to run. Mandatory; at least one action is
  required.

A minimal example for a recipe:

```yaml
architecture: arm64
sectorsize: 512

actions:
  - action: debootstrap
    suite: trixie
```

### Recipe Template Variables

Two kinds of variables are passed to the recipe and they use different syntax:

- Variables passed on the command line with `-t NAME:VALUE` are fields of the
  template data object and are referenced in the recipe as `{{ .NAME }}`.
- Local variables are declared inside the recipe with
  `{{ $name := "value" }}` and referenced as `{{ $name }}`.

The common idiom combines both to give a command-line variable a default:

```yaml
{{- $image := or .image "debian.tgz" -}}

architecture: arm64

actions:
  - action: pack
    file: {{ $image }}
```

Here `debos recipe.yaml` creates `debian.tgz`, while
`debos -t image:custom.tgz recipe.yaml` creates `custom.tgz`.

The `recipe` action passes variables injected with `-t NAME:VALUE` down to the included recipe.
The `architecture` property of an included recipe must match the parent and
is injected automatically as a template variable.

### Recipe Template Functions

In addition to the standard Go template functions and the
[slim-sprig](https://go-task.github.io/slim-sprig/) function library, the
following custom functions are available:

- `escape` -- shell-escape the argument, e.g. `{{ escape $var }}`.
- `uuid5` -- generate a deterministic version 5 UUID,
  `{{ uuid5 $namespace $data }}`. `$namespace` must be a valid UUID. Passing
  fixed values produces a reproducible UUID, which may be useful for the 
  [image-partition](#image-partition-action) action.
- `sector` -- appends an `s` suffix, e.g. `{{ sector 256 }}` becomes `256s`.
  Deprecated: the sector size is unknown at templating time, so write the
  offset directly as `256s` and let the action resolve it from `sectorsize`.

Use `--print-recipe` to print the recipe after templating and `--dry-run`
to compose and verify the recipe without performing any work.

## Recipe Actions

Actions are executed in the order they are listed in the recipe.
Unless `--disable-fakemachine` is given (or no backend is available), all actions
run inside the fakemachine virtual machine. The actions are grouped below by
purpose.

Each action accepts two common properties in addition to the
action-specific ones listed below:

- `action` -- string, mandatory. Selects the action to run; it must be one of
  the actions documented below.
- `description` -- string, optional. Free-text label shown in debos output
  for this action; defaults to the action name.

Only the action-specific properties are listed in each action's section.

The following actions construct the target root filesystem.

### `debootstrap` Action

Construct the target rootfs with the `debootstrap` tool.

Properties:

- `suite` -- string, mandatory. Release code name or symbolic name (e.g.
  `trixie`, `stable`).
- `mirror` -- string, optional, default `http://deb.debian.org/debian`. A
  Debian-compatible repository URL.
- `variant` -- string, optional. debootstrap variant to use, e.g. `minbase`.
- `components` -- list of strings, optional, default `[main]`.
- `keyring-package` -- string, optional. Keyring package validating the
  repository.
- `keyring-file` -- string, optional. Keyring file validating the repository
  (path relative to the recipe directory).
- `certificate` -- string, optional. Client certificate file.
- `private-key` -- string, optional. Client private key file.
- `merged-usr` -- bool, optional, default `true`. Create a merged-`/usr`
  filesystem.
- `check-gpg` -- bool, optional, default `true`. Verify the repository GPG
  signature.

Example:

```yaml
- action: debootstrap
  suite: trixie
  components:
    - main
    - non-free-firmware
  mirror: https://deb.debian.org/debian
  variant: minbase
```

Notes: `/etc/resolv.conf` is removed after the bootstrap because debootstrap
copies it from the host. For a foreign architecture the second stage runs in
the chroot.

### `mmdebstrap` Action

Construct the target rootfs with the `mmdebstrap` tool. Compared to
`debootstrap` it can install extra packages and accept arbitrary apt/dpkg
options during the bootstrap.

Properties:

- `suite` -- string, mandatory.
- `mirrors` -- list of strings, optional. Debian-compatible repository URLs.
- `variant` -- string, optional.
- `components` -- list of strings, optional, default `[main]`.
- `keyring-packages` -- list of strings, optional.
- `keyring-files` -- list of strings, optional. Paths relative to the recipe
  directory.
- `merged-usr` -- bool, optional. If unset the distribution default is used
  (this differs from `debootstrap`, where the default is `true`).
- `include` -- list of strings, optional. Packages to install during the
  bootstrap.
- `dpkg-opts` -- list of strings, optional. Arbitrary dpkg options.
- `apt-opts` -- list of strings, optional. Arbitrary apt options.

Example:

```yaml
- action: mmdebstrap
  suite: trixie
  mirrors:
    - https://deb.debian.org/debian
  variant: minbase
  include:
    - ca-certificates
  apt-opts:
    - -oAcquire::Languages=none
```

Notes: `/etc/resolv.conf` is removed after the bootstrap.

### `pacstrap` Action

Construct an Arch Linux target rootfs with the `pacstrap` tool.

Properties:

- `config` -- string, mandatory. `pacman.conf` file, relative to the recipe
  directory.
- `mirror` -- string, mandatory. mirrorlist file, relative to the recipe
  directory.
- `packages` -- list of strings, optional. If empty only the `base`
  metapackage is installed.

Example:

```yaml
- action: pacstrap
  config: files/pacman.conf
  mirror: files/mirrorlist
  packages:
    - base
    - linux
    - openssh
```

Notes: this action requires fakemachine; it fails with
`--disable-fakemachine`.

The following actions install packages into an existing rootfs.

### `apt` Action

Install packages and their dependencies into the target rootfs with `apt`.

Properties:

- `packages` -- list of strings, mandatory. Packages to install.
- `recommends` -- bool, optional, default `false`. Install recommended
  packages.
- `unauthenticated` -- bool, optional, default `false`. Allow unauthenticated
  packages.
- `update` -- bool, optional, default `true`. Run `apt update` before
  installing.

Example:

```yaml
- action: apt
  recommends: false
  packages:
    - openssh-server
    - sudo
    - systemd-sysv
```

### `pacman` Action

Install packages and their dependencies into an Arch Linux rootfs with
`pacman`.

Properties:

- `packages` -- list of strings, mandatory.

Example:

```yaml
- action: pacman
  packages:
    - vim
    - sudo
```

### `install-deb` Action

Install local `.deb` packages and their dependencies into the target rootfs
using `apt`. Dependencies are resolved from the configured apt sources.

Properties:

- `origin` -- string, optional, default the recipe directory. The named
  origin (see *The origin property*) containing the packages.
- `packages` -- list of strings, optional. `.deb` files or glob patterns
  resolved within `origin`. May be omitted when `origin` refers to a single
  downloaded file.
- `recommends` -- bool, optional, default `false`.
- `unauthenticated` -- bool, optional, default `false`.
- `update` -- bool, optional, default `true`.

Origin behavior: when `origin` is a directory, `packages` selects files
within it (glob patterns allowed). When `origin` is a single file produced by
a `download` action, omit `packages`.

Example -- install local packages shipped with the recipe:

```yaml
- action: install-deb
  packages:
    - pkgs/*.deb
    - pkgs/bmap-tools_*_all.deb
```

Example -- download a `.deb` and install it with its dependencies:

```yaml
- action: download
  url: http://example.com/foo_1.0_all.deb
  name: foo-deb

- action: install-deb
  origin: foo-deb
```

The following actions add files to the target filesystem.

### `download` Action

Download a single file from the internet and optionally unpack it. The
download is registered as a named origin for use by later actions.

Properties:

- `url` -- string, mandatory. `http` or `https` URL.
- `name` -- string, mandatory. Origin name exported for later actions.
- `filename` -- string, optional. Output file name; derived from the URL when
  omitted.
- `unpack` -- bool, optional, default `false`. Extract the archive after
  download.
- `compression` -- string, optional. Compression hint for `unpack`.
- `sha256sum` -- string, optional. Expected SHA256 sum (exactly 64 hex
  characters); verified after download.

Origin behavior: the origin named by `name` resolves to the downloaded file,
or, when `unpack` is `true`, to the extraction directory (`<filename>.d`).

Example:

```yaml
- action: download
  url: https://example.com/rootfs.tar.gz
  name: rootfs
  sha256sum: 0123...cdef
  unpack: true
  compression: gz

- action: overlay
  origin: rootfs
```

### `overlay` Action

Recursively copy a file or directory tree into the target filesystem,
preserving permissions and extended attributes.

Properties:

- `origin` -- string, optional, default the recipe directory. Source origin
  (see *The origin property*).
- `source` -- string, optional. Path relative to `origin`. If empty the
  origin root is used.
- `destination` -- string, optional, default the rootfs root. Absolute path
  inside the target filesystem.

At least one of `source` or `origin` must be set.

Origin behavior: `origin` is resolved through the origin map. When `origin`
is empty or `recipe`, `source` must exist on disk at verification time.

Example:

```yaml
- action: overlay
  source: overlays/sudo
```

This copies `overlays/sudo/etc/sudoers.d/user` from the recipe directory to
`/etc/sudoers.d/user` in the rootfs.

### `unpack` Action

Unpack a (optionally compressed) tar archive into the target filesystem.

Properties:

- `origin` -- string, optional, default the artifact directory. Origin
  containing the archive.
- `file` -- string, optional. Archive file name within `origin`. May be
  omitted when `origin` is a single downloaded file.
- `destdir` -- string, optional, default the rootfs root. Destination path
  inside the target filesystem.
- `compression` -- string, optional. Compression hint; auto-detected from the
  file extension when omitted.

At least one of `file` or `origin` must be set.

Origin behavior: by default the archive is looked up in the artifact
directory; set `origin` to a `download` name to unpack a downloaded archive.

Example:

```yaml
- action: unpack
  origin: artifacts
  file: rootfs.tar.gz
  destdir: /
```

The following actions create and populate the output image.

### `image-partition` Action

Create an image file, partition it and format the filesystems. Mountpoints
can be defined so partitions are mounted during the build and, by default,
mounted at boot in the final system. Mountpoints are sorted by their position
in the filesystem hierarchy, so their order in the recipe does not matter.

Properties:

- `imagename` -- string, mandatory. Output image file, relative to the
  artifact directory.
- `imagesize` -- string, mandatory. Image size in human-readable form, e.g.
  `100MB`, `4GB`.
- `partitiontype` -- string, mandatory. `gpt` or `msdos`.
- `diskid` -- string, optional. Disk identifier; GUID format for `gpt`, a
  32-bit hex value (no dashes) for `msdos`.
- `gpt_gap` -- string, optional. Shift the GPT to leave a gap for a
  bootloader (`gpt` only; requires parted support).
- `partitions` -- list, mandatory. At least one partition (see below).
- `mountpoints` -- list, optional. Mountpoints (see below).

#### partition properties

- `name` -- string, mandatory. Unique; used to reference the partition from
  mountpoints and `raw`.
- `fs` -- string, mandatory. Filesystem type (`ext4`, `btrfs`, `f2fs`, `xfs`,
  `hfs`, `hfsplus`, `vfat`/`fat*`, ...). Use `none` for a partition without a
  filesystem.
- `start` -- string, mandatory. Start offset from the beginning of the disk;
  human readable (`32MB`) or percentage (`0%`).
- `end` -- string, mandatory. End offset; same format as `start` (`100%`).
- `partlabel` -- string, optional, default the partition `name`. GPT
  partition label (`gpt` only).
- `fslabel` -- string, optional, default the partition `name`. Filesystem
  label.
- `parttype` -- string, optional. Partition type; 2-char hex for `msdos`,
  36-char GUID for `gpt`.
- `partattrs` -- list, optional. GPT partition attribute bits.
- `partuuid` -- string, optional. GPT partition UUID (e.g. from `uuid5`).
- `fsuuid` -- string, optional. Filesystem UUID (btrfs, ext2/3/4, xfs).
- `flags` -- list, optional. parted(8) `set` flags, e.g. `[ boot ]`.
- `features` -- list, optional. Extra filesystem features to enable.
- `extendedoptions` -- list, optional. Extra filesystem extended options.
- `fsck` -- bool, optional, default `true`. When `false`, sets `fs_passno` to
  `0` (no boot-time check).

#### mountpoint properties

- `partition` -- string, mandatory. A partition `name` defined above.
- `mountpoint` -- string, mandatory. Unique path in the target filesystem.
- `options` -- list, optional. fstab options for this entry.
- `buildtime` -- bool, optional, default `false`. Mount only during the debos
  run; no `/etc/fstab` entry is created and the directory is removed from the
  image.

Example -- two-partition msdos image (FAT firmware + ext4 root):

```yaml
- action: image-partition
  imagename: debian-rpi3.img
  imagesize: 1GB
  partitiontype: msdos
  mountpoints:
    - mountpoint: /
      partition: root
    - mountpoint: /boot/firmware
      partition: firmware
      options: [ x-systemd.automount ]
  partitions:
    - name: firmware
      fs: vfat
      start: 0%
      end: 64MB
    - name: root
      fs: ext4
      start: 64MB
      end: 100%
      flags: [ boot ]
```

The partition device paths are recorded by name so the `raw` action can
write to a specific partition.

### `filesystem-deploy` Action

Deploy the prepared root filesystem onto the image created by a previous
`image-partition` action, and optionally generate `/etc/fstab` and the kernel
command line.

Properties:

- `setup-fstab` -- bool, optional, default `true`. Generate `/etc/fstab` from
  the partition layout.
- `setup-kernel-cmdline` -- bool, optional, default `true`. Add the root
  partition to `/etc/kernel/cmdline`.
- `append-kernel-cmdline` -- string, optional. Extra kernel command line
  arguments.

Example:

```yaml
- action: filesystem-deploy
  setup-fstab: true
  setup-kernel-cmdline: false
```

Notes: a previous `image-partition` action is required. Afterwards the build
operates on the mounted image (`context.Rootdir` and the `filesystem` origin
point at the mounted image), so later actions modify the image directly.

### `raw` Action

Directly write a file into the output image at a given offset. Typically used
for bootloaders.

Properties:

- `source` -- string, mandatory. File to write into the image.
- `origin` -- string, optional, default the recipe directory. Origin
  containing `source`.
- `offset` -- string, optional, default `0`. Offset in bytes, or in sectors
  with an `s` suffix (e.g. `256s`); the sector size comes from the recipe
  `sectorsize` header.
- `partition` -- string, optional. Named partition to write to; if unset the
  whole image is used.

Origin behavior: `origin` is resolved through the origin map. The deprecated
`path` property and the deprecated `{{ sector N }}` template function still
work but should be replaced by `origin`/`source` and a literal `Ns` offset.

Example:

```yaml
- action: raw
  origin: recipe
  source: files/u-boot.bin
  offset: 8192s
```

The following actions integrate OSTree.

### `ostree-commit` Action

Create an OSTree commit from the root filesystem.

Properties:

- `repository` -- string, mandatory. OSTree repository path, relative to the
  artifact directory.
- `branch` -- string, mandatory. OSTree branch name.
- `subject` -- string, optional. One-line commit description.
- `collection-id` -- string, optional. Collection ID for ref binding
  (requires libostree 2018.6+).
- `ref-binding` -- list of strings, optional. Branches the commit must be
  retrieved from. Defaults to `[branch]` when `collection-id` is set.
- `metadata` -- map of string to string, optional. Commit metadata.

Example:

```yaml
- action: ostree-commit
  repository: repo
  branch: debian/trixie
  subject: Debian trixie
```

### `ostree-deploy` Action

Deploy an OSTree branch into the image.

Properties:

- `repository` -- string, mandatory. Local OSTree repository, relative to the
  artifact directory.
- `remote_repository` -- string, mandatory. URL of the remote OSTree
  repository (remote pull is currently not implemented; use a local
  repository).
- `branch` -- string, mandatory. Branch to deploy.
- `os` -- string, mandatory. OSTree OS deployment name.
- `setup-fstab` -- bool, optional, default `true`.
- `setup-kernel-cmdline` -- bool, optional, default `true`.
- `append-kernel-cmdline` -- string, optional.
- `tls-client-cert-path` -- string, optional. Client certificate for the
  remote repository.
- `tls-client-key-path` -- string, optional. Client certificate key.
- `collection-id` -- string, optional. Collection ID ref binding (requires
  libostree 2018.6+).

Example:

```yaml
- action: ostree-deploy
  repository: repo
  remote_repository: http://localhost/repo
  branch: debian/trixie
  os: debian
```

Notes: a previous `image-partition` action is required.

The following action archives the root filesystem.

### `pack` Action

Create a tarball from the root filesystem.

Properties:

- `file` -- string, mandatory. Output tarball, relative to the artifact
  directory.
- `compression` -- string, optional, default `gz`. One of `bzip2`, `gz`,
  `lzip`, `lzma`, `lzop`, `xz`, `zstd`, `auto`, `none`.
- `subdir` -- string, optional. Subdirectory within the rootfs to archive
  (must exist); defaults to the whole rootfs.

Example:

```yaml
- action: pack
  file: rootfs.tar.gz
  compression: gz
```

Notes: `pigz` is used automatically for `gz` compression when available.

The following actions control execution flow.

### `recipe` Action

Include the actions of another recipe, optionally overriding template
variables.

Properties:

- `recipe` -- string, mandatory. Path to the recipe to include (relative to
  the current recipe directory).
- `variables` -- map of string to string, optional. Template variables passed
  to the included recipe.

Example:

```yaml
- action: recipe
  recipe: partition.yaml
  variables:
    imagesize: 4GB
```

Notes: the included recipe's `architecture` must match the parent and is
injected automatically. Only one `image-partition` action may exist across
the combined recipes.

### `run` Action

Run a command or a script, either in the build host environment (the
fakemachine) or in the target filesystem.

Properties:

- `command` -- string. A command line, or -- using a YAML block scalar
  (`command: |`) -- a multi-line shell script. The command(s) must be
  available in the host or chroot environment. Mutually exclusive with
  `script`.
- `script` -- string. Script with arguments; the script must be located in
  the recipe directory. Mutually exclusive with `command`.
- `chroot` -- bool, optional, default `false`. Run inside the target
  filesystem instead of the build host.
- `postprocess` -- bool, optional, default `false`. Run after all other
  actions, with the artifact directory as working directory. Mutually
  exclusive with `chroot`.
- `label` -- string, optional. Label used for output; derived from the
  command/script (truncated at 40 characters) when empty.

One of `command` or `script` is required. The command or script is run with
root privileges via `sh -e -c`. Because `command` is passed to the shell as a
single string, a YAML block scalar (`command: |`) may contain several commands
on separate lines: they run as one script, the first line is used as the
output label, and -- because of the `-e` flag -- execution stops at the first
command that fails.

Environment variables: debos exports the following variables to non-chroot
`run` actions. Inside a chroot the command runs in the target's own
environment and debos does not inject these (a `script` is bind-mounted to
`/tmp/script`). `postprocess` runs outside the chroot.

| Variable    | host (`chroot: false`) | `chroot: true`       | `postprocess: true` |
|-------------|------------------------|----------------------|---------------------|
| RECIPEDIR   | yes                    | no                   | yes                 |
| ARTIFACTDIR | yes                    | no                   | yes                 |
| ROOTDIR     | yes                    | no                   | no                  |
| IMAGEMNTDIR | yes (if image mounted) | no                   | no                  |
| IMAGE       | yes (if image exists)  | yes (if image exists)| no                  |

Example -- run a script from the recipe inside the chroot:

```yaml
- action: run
  chroot: true
  script: scripts/setup-user.sh
```

Example -- build a squashfs of the rootfs on the host:

```yaml
- action: run
  chroot: false
  command: mksquashfs ${ROOTDIR} ${ARTIFACTDIR}/root.squashfs -comp xz -e rwdata/
```

Example -- run several commands as a single script with `command: |`:

```yaml
- action: run
  chroot: true
  command: |
    apt-get update
    apt-get -y dist-upgrade
    rm -rf /var/lib/apt/lists/*
```

Example -- compress the finished image as a postprocessing step:

```yaml
{{- $image := or .image "debian.img" -}}
- action: run
  postprocess: true
  command: gzip -f {{ $image }}
```

## Environment variables

### Host-propagated variables

debos reads a predefined list of environment variables from the host and
propagates them to the fakemachine build environment. The list is defined by
`environVars` in `cmd/debos/debos.go` and currently covers the proxy
variables documented at
https://wiki.archlinux.org/index.php/proxy_settings:

```
http_proxy, https_proxy, ftp_proxy, rsync_proxy, all_proxy, no_proxy
```

For each of the above variables, both the lower and upper case variant is probed
on the host and, if found, propagated to fakemachine preserving the case.
So, if the host defines `HTTP_PROXY` and `no_proxy`, both are propagated as-is.

### Setting and Overriding Variables

The `--environ-var` and `-e` options specify, overwrite or unset environment
variables for fakemachine:

```bash
debos -e ENVIRONVAR:VALUE ...
```

To unset a variable (e.g. prevent it from being propagated to fakemachine) use the
same syntax without a value. Multiple `-e` options may be given.

### Proxy configuration caveats

While the proxy variables are exported from the host to fakemachine, there
are two known sources of issues:

- `localhost` will not work from fakemachine. Use an address that is valid on
  your network. debos warns if a variable contains `localhost`, `127.0.0.1`
  or `::1`.
- Applications and scripts run inside fakemachine may expect differently
  named or differently cased proxy variables; check which variables they use.

### Variables exposed to the run action

The `run` action exposes `RECIPEDIR`, `ARTIFACTDIR`, `ROOTDIR`,
`IMAGEMNTDIR` and `IMAGE` to non-chroot commands and scripts. Which variables
are set depends on `chroot` and `postprocess`; see the table in the *run*
action above.

### Origins

Several actions accept an `origin` property that selects where their input
comes from. The following origins are always available:

- `recipe` -- the directory the recipe is in.
- `filesystem` -- the target filesystem root. Set by `filesystem-deploy` and
  `ostree-deploy` to the mounted image after deployment.
- `artifacts` -- the artifact directory.

In addition, each `download` action registers a named origin (its `name`
property) that resolves to the downloaded file, or to the extraction
directory when `unpack` is enabled. Origins are resolved at runtime, so a
`download` (or `filesystem-deploy`) earlier in the recipe makes its origin
available to later actions.

## Fakemachine Backend

debos (unless running debos with the `--disable-fakemachine` argument)
creates and spawns a virtual machine using [fakemachine](https://github.com/go-debos/fakemachine)
and executes the actions defined by the recipe inside the virtual machine.
This helps ensure recipes are reproducible no matter the host environment.

Fakemachine can use different virtualisation backends to spawn the virtual
machine, for more information see the [fakemachine documentation](https://github.com/go-debos/fakemachine).

By default the backend will automatically be selected based on what is
supported by the host machine, but this can be overridden using the
`--fakemachine-backend` / `-b` option. If no backends are supported,
debos reverts to running the recipe on the host without creating a
fakemachine.
